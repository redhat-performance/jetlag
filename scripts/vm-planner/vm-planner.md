# VM Planner

`vm-planner.py` generates Ansible `[hv_vm]` inventory records for Jetlag VMNO/hybrid deployments. It reads hypervisors from an existing inventory `[hv]` section, applies a topology definition (VM hardware profiles and cluster composition), and writes `[hv_vm]` host entries compatible with `hv-vm-create.yml`.

## Quick start

```bash
# Copy and edit topology
cp scripts/vm-planner/topology.json.sample scripts/vm-planner/topology.json

# Preview planned inventory records (no files modified)
python3 scripts/vm-planner/vm-planner.py \
  --dry-run \
  --inventory-file ansible/inventory/cloud12.local \
  --vm-types scripts/vm-planner/topology.json \
  --cluster-type standard

# Check hypervisor capacity before generating inventory
python3 scripts/vm-planner/vm-planner.py \
  --dry-run \
  --assess-hv \
  --inventory-file ansible/inventory/cloud12.local \
  --vm-types scripts/vm-planner/topology.json \
  --cluster-type standard

# Generate a new inventory file (default)
python3 scripts/vm-planner/vm-planner.py \
  --inventory-file ansible/inventory/cloud12.local \
  --vm-types scripts/vm-planner/topology.json \
  --cluster-type standard
# -> creates ansible/inventory/cloud12-1standard-3m-2w.local
```

## Prerequisites

- Python 3
- An Ansible inventory file with a populated `[hv]` section (generated by Jetlag `create-inventory.yml`)
- `sshpass` (only when using `--assess-hv`)

## Topology file

Topology defines **what VMs look like** (`vm_types`) and **what clusters to deploy** (`cluster_types`). Copy `topology.json.sample` and customize.

Supported formats: **JSON** (`.json`), **YAML** (`.yml`/`.yaml`), **Python** (`.py` with `VM_TYPES` / `CLUSTER_TYPES`).

### `vm_types`

Hardware/network template per VM role. Used for every VM of that type.

| Field | Required | Description |
|-------|----------|-------------|
| `per_hv` | Yes* | Max VMs of this type per hypervisor when using `--spread` (*VM type named `sno` always assumes 1) |
| `cpus` | Yes | vCPU count |
| `memory` | Yes | Memory in GB |
| `disk_size` | Yes | Disk size in GB |
| `bw_avg` | Yes | Average bandwidth limit |
| `bw_peak` | Yes | Peak bandwidth limit |
| `bw_burst` | Yes | Burst bandwidth limit |
| `disk_location` | Yes | Default libvirt image path on the hypervisor |
| `misc` | No | Extra `key=value` fields appended to each inventory record |

Example `misc` block:

```json
"misc": {
  "vendor": "Libvirt",
  "install_disk": "/dev/sda"
}
```

### `cluster_types`

Defines cluster layouts and how many to deploy. Only **one** cluster type is selected per run via `--cluster-type`. To deploy multiple clusters with the same layout, increase `count`. For a non-standard mix of VM roles, define a custom cluster type in the topology.

| Field | Required | Description |
|-------|----------|-------------|
| `count` | Yes | Number of clusters of this type |
| `vms` | Yes | Map of VM type name → count per cluster |

Example:

```json
"standard": {
  "count": 1,
  "vms": {
    "master": 3,
    "worker": 2
  }
}
```

Custom cluster type for a mixed layout:

```json
"hybrid-lab": {
  "count": 2,
  "vms": {
    "master": 3,
    "worker": 2,
    "gpu": 1
  }
}
```

```bash
python3 scripts/vm-planner/vm-planner.py \
  --inventory-file ansible/inventory/cloud12.local \
  --vm-types scripts/vm-planner/topology.json \
  --cluster-type hybrid-lab
# -> cloud12-2hybrid-lab-3m-2w-1g.local
```

To add VMs from a different cluster type to an existing inventory, run the planner again with `--append` and a different `--cluster-type`. See [Important: `--append` addressing](#important-append-addressing) — you must adjust start parameters to avoid conflicts.

## Placement modes

### Default (cluster-local)

Each cluster is placed on a **single hypervisor**. Multiple clusters each get their own HV when possible. If there are more clusters than hypervisors, clusters are assigned round-robin.

Use `--start-at-hv N` to skip the first hypervisors in `[hv]` and begin placement at the Nth host (0-based: `0` is the first hypervisor, `1` is the second). Clusters still rotate round-robin from that starting point. Useful when earlier hypervisors already host VMs from a prior run.

### `--spread`

Each cluster's VMs are spread across hypervisors by VM type, honoring each type's `per_hv` limit (place up to `per_hv` VMs on one HV before moving to the next). Cluster metadata is preserved for naming and comments.

**SNO clusters** always use `per_hv=1` in spread mode, regardless of topology.

`--start-at-hv` applies in spread mode as well: the first cluster begins its HV rotation at the specified hypervisor.

## VM naming

| Mode | Naming |
|------|--------|
| Default | `vm00001`, `vm00002`, … |
| `--vm-prefix node` | `node00001`, `node00002`, … |
| `--organize-vm-names`* | `cluster-0-master-1`, `cluster-0-worker-2`, … |
| * `--organize-vm-names` + SNO cluster | `sno-0`, `sno-1`, … (0-based among SNO clusters only) |

`--organize-vm-names` and `--vm-prefix` are mutually exclusive in effect; organized naming takes precedence when `--organize-vm-names` is set.

## Output inventory files

### Default (new file)

By default the script creates a **new** inventory file next to the input file. The original is not modified.

Filename pattern:

```
{original_name}-{count}{cluster type}-{count}{vm type letter}-....local
```

The VM composition segment is built from each VM type in the cluster `vms` map: `{count}{first letter of type name}`, joined with `-`. Types appear in the order defined in the topology. Zero-count types are omitted. The VM segment reflects **per-cluster** composition; the cluster `count` is in the cluster segment.

| Cluster composition | VM segment |
|---------------------|------------|
| `master: 3, worker: 2` | `3m-2w` |
| `master: 3` | `3m` |
| `master: 1` (SNO) | `1m` |

Examples:

- `cloud12.local` + `standard` (3 masters, 2 workers) → `cloud12-1standard-3m-2w.local`
- `cloud12.local` + `compact` (3 masters) → `cloud12-1compact-3m.local`
- `cloud12.local` + `hybrid-lab` x2 (3 masters, 2 workers, 1 gpu) → `cloud12-2hybrid-lab-3m-2w-1g.local`

### `--overwrite-original`

Write `[hv_vm]` records into the input `--inventory-file` instead of creating a derived file.

### `--append`

Append new `[hv_vm]` records to an existing file instead of replacing the section.

| Scenario | Behavior |
|----------|----------|
| `--append --overwrite-original` | Append to the original inventory |
| `--append` + derived file exists | Append to the derived file |
| `--append` + derived file does not exist | No effect; a new file is created (message printed) |

#### Important: `--append` addressing {#important-append-addressing}

> **Important**
>
> `--append` only adds lines to the inventory file. It does **not** scan existing records or continue numbering automatically. Each run starts from the default `--start-vm-id`, `--start-mac`, and `--start-vnc` values unless you set them explicitly.
>
> If you append without adjusting these, new VMs will **duplicate** values already in use:
>
> | Field | Controlled by | Conflict when defaults are reused |
> |-------|---------------|-----------------------------------|
> | VM name / UUID | `--start-vm-id` | Same `vm00001`, same `domain_uuid` |
> | IP address | `--start-vm-id` + `--vm-ip-offset` | Same `ip=` as an existing VM |
> | MAC address | `--start-mac` + VM ID | Same `mac_address=` |
> | VNC port | `--start-vnc` (per hypervisor) | Same `vnc_port=` on the same HV |
>
> Before appending, set start values to continue after the last existing VM. Example: after 54 VMs, append 3 more with `--start-vm-id 55` and `--start-mac 52:54:00:00:00:37`. If new VMs share hypervisors with existing ones, also set `--start-vnc` high enough to avoid VNC port collisions on those HVs.
>
> `--append` is incompatible with `--sub-groups` and with inventories that use `[hv_vm:children]`.

## Generated record format

Each `[hv_vm]` line includes Jetlag-compatible fields:

- `ansible_host`, `hv_ip`, `ip`, `cpus`, `memory`, `disk_size`, `vnc_port`
- `mac_address` — QEMU/KVM prefix (`52:54:00:…`), deterministic per VM number
- `domain_uuid` — Ansible-compatible namespaced UUID (matches Jetlag `to_uuid`)
- `disk_location`, `bw_avg`, `bw_peak`, `bw_burst`
- Optional `misc` fields from topology
- Optional `role={vm type}` when using `--tag-roles`

Cluster blocks are separated in the inventory with comment headers:

```
# cluster 0 (standard): 3 master, 2 worker
vm00001 ansible_host=...
...

# cluster 1 (compact): 3 master
vm00006 ansible_host=...
```

## CLI reference

### Required

| Option | Description |
|--------|-------------|
| `--inventory-file PATH` | Source Ansible inventory with a `[hv]` section. Hypervisor `ip=` and optional `disk2_enable=` are read from host entries. `controlplane_network` is read from `[all:vars]` when present. |

### Topology and clusters

| Option | Default | Description |
|--------|---------|-------------|
| `--vm-types PATH` | Built-in defaults | Topology file (JSON, YAML, or Python) with `vm_types` and `cluster_types` |
| `--cluster-type NAME` | `vmno` | Cluster type to deploy. One type per run; must exist in `cluster_types` |

### Output control

| Option | Default | Description |
|--------|---------|-------------|
| `--dry-run` | off | Print a summary and exact `[hv_vm]` lines without writing files |
| `--short` | off | With `--dry-run`, print only the `[hv_vm]` section (no summary) for copy-paste |
| `--overwrite-original` | off | Modify `--inventory-file` in place instead of creating a derived file |
| `--append` | off | Append records to an existing inventory file instead of replacing `[hv_vm]` entries |

### Hypervisor assessment

| Option | Default | Description |
|--------|---------|-------------|
| `--assess-hv` | off | SSH to each hypervisor and compare planned VM load against live CPU, memory, and disk capacity |

Works with or without `--dry-run`. When generating inventory (without `--dry-run`), the assessment is printed before writing files.

Connects over SSH using `ansible_user` and `ansible_ssh_pass` from `[hv:vars]` (per-host overrides supported). Requires `sshpass` on the machine running vm-planner.

The **Hypervisors** section in the output shows planned load and queried capacity:

```
Hypervisors (3)
------------------------------------------------------------
  hv01 (198.18.0.6)  disk2=True
    planned: 48 CPU, 128 GB RAM, 500 GB disk (5 VMs)
    capacity: 32 CPU, 256 GB RAM, disk 400 GB (/var/lib/libvirt/images), 2000 GB (/mnt/disk2/libvirt/images)
    CPU overcommit: 1.50x (48 vCPUs planned, 32 CPUs available)
    WARNING: memory exceeds capacity (200 GB planned, 128 GB available)
    Disk overcommit on /var/lib/libvirt/images: 1.25x (500 GB planned, 400 GB available)
```

When planned use exceeds available resources:

| Resource | Output |
|----------|--------|
| CPU | Overcommit ratio (e.g. `CPU overcommit: 1.50x (48 vCPUs planned, 32 CPUs available)`) |
| Memory | Warning (e.g. `WARNING: memory exceeds capacity (200 GB planned, 128 GB available)`) |
| Disk | Per-path overcommit ratio based on VM `disk_location` placement |

Disk planning follows the same rules as inventory generation: the first `--default-disk-vms` VMs per hypervisor use the default libvirt path; overflow uses `--disk2-mount-path` when `disk2_enable=True`.

### Short dry-run output (`--short`)

Use with `--dry-run` to print **only** the `[hv_vm]` section — no summary, placement details, or footer. Intended for copy-paste into an inventory file.

Cannot be combined with `--assess-hv`.

```bash
python3 scripts/vm-planner/vm-planner.py \
  --dry-run \
  --short \
  --inventory-file ansible/inventory/cloud12.local \
  --vm-types scripts/vm-planner/topology.json \
  --cluster-type standard
```

Example output:

```
[hv_vm]
# cluster 0 (standard): 3 master, 2 worker
vm00001 ansible_host=...
vm00002 ansible_host=...
...
```

### Placement and naming

| Option | Default | Description |
|--------|---------|-------------|
| `--spread` | off | Spread each cluster's VMs across hypervisors using `per_hv` limits |
| `--start-at-hv N` | `0` | 0-based index of the first hypervisor in `[hv]` to use; rotates round-robin from there |
| `--organize-vm-names` | off | Use `cluster-{n}-{type}-{instance}` names; SNO clusters use `sno-{index}` |
| `--vm-prefix PREFIX` | `vm` | Custom prefix for sequential names (e.g. `node` → `node00001`) |
| `--tag-roles` | off | Add `role={vm type}` to each record (e.g. `role=master`, `role=worker`) |
| `--sub-groups` | off | Place each cluster in `[vm_cluster_{type}_{index}]` under `[hv_vm:children]` |

`--sub-groups` cannot be used with `--append`. To update an inventory generated with sub-groups, re-run without `--append`.

Without `--sub-groups`, an input file using `[hv_vm:children]` is converted back to a flat `[hv_vm]` section.

### Sub-group inventory layout (`--sub-groups`)

Each cluster is placed in its own group. `[hv_vm]` is replaced by a parent group:

```ini
[hv_vm:children]
vm_cluster_standard_0
vm_cluster_standard_1

[vm_cluster_standard_0]
vm00001 ansible_host=...
vm00002 ansible_host=...

[hv_vm:vars]
...
```

Group names use a 0-based cluster index: `vm_cluster_{cluster type}_{index}`.

### VM addressing and disks

| Option | Default | Description |
|--------|---------|-------------|
| `--vm-ip-offset N` | `20` | Host offset within `controlplane_network` for VM IPs |
| `--start-vm-id N` | `1` | First VM numeric ID (`1` → `vm00001`) |
| `--start-mac ADDR` | `52:54:00:00:00:01` | Starting MAC address (sequential from this value) |
| `--start-vnc PORT` | `5901` | Starting VNC port per hypervisor |
| `--default-disk-vms N` | `3` | VMs per HV on the default disk before overflow to disk2 |
| `--disk2-mount-path PATH` | `/mnt/disk2` | Disk2 image path when `disk2_enable=True` on a hypervisor |

## Workflow with Jetlag

1. Run `create-inventory.yml` to generate an inventory with `[hv]` hosts
2. Customize `topology.json` for your cluster layout and VM sizing
3. Run `vm-planner.py` with `--dry-run` to verify placement; add `--assess-hv` to check hypervisor capacity
4. Use the generated inventory with `hv-vm-create.yml` and deployment playbooks

## Error handling

The script prints a single user-friendly error message to stderr and exits with code `1`. Common issues:

- Missing topology or inventory file
- Invalid JSON in topology
- Unknown `--cluster-type`
- Multiple cluster types passed to `--cluster-type`
- Missing `ansible_user` / `ansible_ssh_pass` when using `--assess-hv`
- `sshpass` not installed when using `--assess-hv`
- `--short` without `--dry-run`, or `--short` with `--assess-hv`
- `--sub-groups` with `--append`
- `--append` on inventories that use `[hv_vm:children]`
- No hypervisors in `[hv]` section
- `--organize-vm-names` combined with invalid topology references
